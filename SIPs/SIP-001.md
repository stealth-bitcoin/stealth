# SIP-001: Stealth — Bitcoin UTXO Privacy Audit Framework

## Introduction

### Abstract

This document specifies the architecture, vulnerability taxonomy, detection algorithms, scoring model, and deployment roadmap for **Stealth**, an open-source privacy audit framework for Bitcoin wallets. Stealth analyzes the transaction history and UTXO set derived from wallet descriptors, surfaces privacy vulnerabilities using on-chain heuristics, and scores wallet privacy posture with a single Privacy Exposure Score (0–100). The goal is to evolve Stealth from a proof-of-concept into a production-grade tool that ships as a standalone application, an embeddable library, and a set of integrations for the broader Bitcoin wallet ecosystem.

Stealth is **read-only**. It holds no keys. It makes no network connections beyond the user's own Bitcoin full node. It is a privacy X-ray, not a wallet.

### Motivation

Bitcoin's transparency is a double-edged sword. Every transaction is publicly auditable, which makes the protocol trustless but also makes financial privacy an active problem users must solve. Chain analysis firms exploit well-documented heuristics — common input ownership, change detection, dust attacks, script-type fingerprinting — to cluster addresses into entities with devastating accuracy.

Today, the average Bitcoin user has no practical way to audit their own wallet's privacy posture. Sparrow Wallet provides coin control. CoinJoin coordinators offer mixing. Nobody offers a **diagnostic** tool that tells a user: "here is what a chain analyst can already infer about you, and here is what you should do about it."

Stealth fills this gap.

### Problem Statement

1. **No defensive tooling exists for regular users.** Chain analysis is a billion-dollar industry. The tools on the other side of the table are either academic papers or commercial forensics software (Chainalysis, Elliptic, Crystal). Users have nothing equivalent pointing inward.
2. **Wallet developers lack a privacy scoring library.** BDK, bitcoinjs-lib, and other wallet SDKs offer coin selection but no built-in privacy analysis. Wallet developers who want to warn users about risky spending patterns must build heuristic engines from scratch.
3. **Self-hosted node operators want privacy dashboards.** Umbrel and Start9 users already run full nodes. A one-click privacy audit app that connects to their local Bitcoin Core instance is a natural fit.
4. **Privacy hygiene is invisible.** Users cannot see the damage done by address reuse, dust merging, or consolidation transactions until it is too late. Stealth makes this damage visible before more harm is done.

### Prior Art and Differentiation

- **OXT Research / Samourai Wallet**: Published foundational heuristic research. Samourai was seized in April 2024. OXT.me remains available but is a hosted block explorer, not a self-hosted audit tool.
- **BlockSci (Princeton)**: Academic blockchain analysis framework requiring full chain parsing; oriented toward researchers, not end users.
- **Chainforensics**: Docker-based UTXO forensics tool for Umbrel. Offense-oriented. Stealth is the inverse: it tells you what forensics tools would find, so you can fix it.
- **btc-heuristics**: MIT-licensed heuristics library for UTXO wallet health scoring, aligned with Maelstrom Grant. Library-only scope; no UI, no integration story, no remediation engine.
- **Open Bitcoin Privacy Project**: Inactive since ~2018. Contributed CoinJoin Sudoku and wallet privacy ratings. Stealth carries the torch with modern heuristics and a production deployment model.

Stealth's differentiation: **defense-first**, **self-hosted**, **descriptor-native**, **integration-ready**, and **remediation-aware**.

## Design

### Architecture Overview

Stealth is structured as a layered system separating analysis logic from I/O, enabling embedding in wallet SDKs without node-connectivity dependencies.

```
+---------------------------------------------------------+
|                    User Interfaces                      |
|  Web UI | CLI | Sparrow Plugin | Wallet SDK callback    |
+---------------------------------------------------------+
|                     REST API                            |
|  /scan  /score  /preflight  /export                     |
+---------------------------------------------------------+
|                  Analysis Engine                        |
|  Descriptor Parser | UTXO Scanner | Heuristic Detectors |
|  Cluster Graph     | Privacy Scorer                     |
+---------------------------------------------------------+
|                  Node Adapter Layer                     |
|  Bitcoin Core RPC  |  (Electrum Protocol — future)      |
+---------------------------------------------------------+
|                  Bitcoin Full Node                      |
|  Bitcoin Core with txindex=1                           |
+---------------------------------------------------------+
```

**Crate boundary**:

```
stealth-core    Pure Rust library — detector logic, scoring, cluster graph, no I/O
stealth-node    Bitcoin Core RPC client, labeled-address DB loader, CLI entrypoint
```

`stealth-core` is the integration target for wallet SDKs (BDK, Sparrow). It has no dependency on node connectivity and can be embedded with zero FFI overhead in any Rust-based wallet.

### Core Rewrite: Python to Rust

The current `detect.py` proof-of-concept is rewritten in Rust as `stealth-core`. Rationale:

- **BDK interoperability.** BDK is Rust-native. A Rust core can be consumed as a crate dependency inside BDK-based wallets with zero overhead.
- **Cross-platform FFI.** Rust compiles to C-compatible shared libraries, enabling bindings for Python (PyO3), Java/Kotlin (JNI), Swift, and WASM.
- **Ecosystem alignment.** rust-bitcoin, rust-miniscript, bdk_chain, and Floresta are all Rust.
- **Embedded and mobile targets.** Python cannot be compiled to WASM or embedded in mobile apps.

The Python implementation is retained as a reference and used to generate ground-truth test vectors for Rust port correctness.

### Inputs and Outputs

#### Inputs

- **Wallet descriptor** (primary). BIP32/44/49/84/86 derivation. Stealth derives addresses and scans the chain for matching transactions.
- **UTXO set** (explicit outpoint list). For auditing a specific set of coins without a descriptor.
- **PSBT** (future: pre-flight analysis). Before broadcasting, Stealth can analyze an unsigned PSBT and warn about privacy leaks the transaction would create. See §PSBT Pre-Flight.
- **Private key** (optional). Only for exploring UTXOs behind hardened derivation paths. Never stored, never transmitted, used in-memory only.

#### Outputs

- **Per-UTXO finding list**: each finding includes ID, severity, confidence, temporality, adversary relevance, evidence, and dependency context.
- **Wallet-level finding list**: findings spanning multiple UTXOs or transactions.
- **Privacy Exposure Score** (0–100): single wallet-level benchmark of current UTXO privacy state.
- **Cluster map**: address-cluster graph inferred from CIOH analysis across the wallet's transaction history (available when `txindex=1`).

### Security Model

**Stealth makes exactly one class of network connection: to a Bitcoin full node over RPC.**  No telemetry. No analytics. No external APIs. No DNS lookups beyond what the OS resolver does for the configured node address. If the node Stealth connects to can be trusted with on-chain queries, Stealth adds no additional attack surface.

**`txindex=1` requirement**: Full-grade analysis of `CLUSTER_MERGE` and `PEELING_CHAIN` requires a full transaction index. Stealth detects at startup whether `txindex` is available and degrades gracefully — all other detectors run without it, with a prominent warning in the report.

Alternative compact nodes (Floresta, Neutrino) are not currently supported; they do not maintain the full transaction index required. Floresta compatibility via its Electrum interface is tracked as a future research item.

### Vulnerability Taxonomy

#### Finding Structure

Every finding carries the following fields:

```
id:                  string identifier (e.g., CIOH)
severity:            CRITICAL | HIGH | MEDIUM | LOW
confidence:          CERTAIN | PROBABLE | POSSIBLE
temporality:         HISTORICAL | ONGOING_PATTERN | ACTIVE_RISK
adversary_relevance: [passive_observer, kyc_exchange, state_adversary]
implies:             list of finding IDs this finding causes (→)
specializes:         finding ID this finding is a specific case of (⊂)
generalizes:         finding ID this finding subsumes (⊃)
```

**Adversary tiers**:
- `passive_observer`: anyone reading the chain with standard heuristics
- `kyc_exchange`: exchanges or services with identity data linked to addresses
- `state_adversary`: intelligence-grade analysis, full transaction graph access

**Temporality**:
- `HISTORICAL`: damage is done; the on-chain event cannot be undone
- `ONGOING_PATTERN`: behavioral finding; stopping the behavior improves future transactions
- `ACTIVE_RISK`: unspent UTXO presents prospective risk; addressable before next spend

#### Dependency Edge Semantics

Three edge types govern how findings relate and how scoring deduplication works:

| Edge | Meaning | Scoring rule |
|------|---------|--------------|
| `A ⊂ B` (A specializes B) | A is a more specific case of B | When A fires alongside B: keep A (specific), suppress B (general) |
| `A ⊃ B` (A generalizes B) | A captures everything B does plus more | When A fires alongside B: keep A (richer), suppress B (subsumed) |
| `A → B` (A causes B) | A's behavior triggers B | Count both — causal, not duplicative |

In both `⊂` and `⊃` cases: **suppress the less-informative finding, always keep A**.

#### Dependency Graph

```
CLUSTER_MERGE (⊃) ──────────────────── CIOH ◄(⊂)── CONSOLIDATION
                                          ▲
                           DUST_SPENDING ─┤(→)
                       UNNECESSARY_INPUT ─┤(⊂)
                       TAINTED_UTXO_MERGE ┘(→)

FEE_RATE_FINGERPRINT        (→)──►
AMOUNT_PATTERN_FINGERPRINT  (→)──► BEHAVIORAL_FINGERPRINT (rollup, fires when ≥2 sub-signals fire)
RBF_FINGERPRINT             (→)──►
OUTPUT_ORDERING_FINGERPRINT (→)──►

ADDRESS_REUSE (⊃) ◄── CHANGE_REUSE (⊂)
```

`BEHAVIORAL_FINGERPRINT` sub-signals use `→` (causal) edges, not `⊂`. The rollup emerges from co-occurrence of behavioral signals; it is not a generalization of any individual signal. Sub-signals and the rollup are both scored independently.

#### Finding Catalog

---

##### Tier 0 — Address-Level

**`ADDRESS_REUSE`**
- Severity: HIGH | Confidence: CERTAIN | Temporality: ONGOING_PATTERN
- Adversary: passive_observer, kyc_exchange, state_adversary
- An address received funds in more than one transaction, linking payment history and balances across senders.

**`CHANGE_REUSE`**
- Severity: CRITICAL | Confidence: CERTAIN | Temporality: ONGOING_PATTERN
- Adversary: passive_observer, kyc_exchange, state_adversary
- `⊂ ADDRESS_REUSE`
- The change output of a spend was sent to an address that previously received external funds. More severe than generic address reuse: it directly links the spender's identity to a prior recipient, collapsing two otherwise separate transaction histories into one.

**`DUST`**
- Severity: LOW | Confidence: CERTAIN | Temporality: HISTORICAL
- Adversary: state_adversary
- UTXO value is below the dust threshold for its script type. Thresholds are derived from `3 × (input_size + output_size) × dust_relay_fee_rate` at the default 3 sat/vbyte relay rate: P2PKH: 546 sat, P2WPKH: 294 sat, P2TR: 330 sat. Dust UTXOs are typically probe transactions used for address-linkage attacks.

---

##### Tier 1 — Single-Transaction

**`CIOH`** (Common Input Ownership Heuristic)
- Severity: CRITICAL | Confidence: PROBABLE | Temporality: HISTORICAL
- Adversary: passive_observer, kyc_exchange, state_adversary
- A transaction has ≥ 2 inputs and the wallet owns at least 2 of them. Standard blockchain analysis treats all inputs as belonging to the same entity.
- **CoinJoin exception**: suppressed if the transaction has ≥ 3 equal-value outputs AND ≥ 3 distinct non-wallet inputs — likely a coordinated mix.

**`CONSOLIDATION`**
- Severity: HIGH | Confidence: CERTAIN | Temporality: HISTORICAL
- Adversary: passive_observer, kyc_exchange
- `⊂ CIOH`
- CIOH fires AND the transaction has ≥ 3 inputs AND ≤ 2 outputs. Explicit consolidation transaction; merges UTXO history into a single traceable coin.

**`UNNECESSARY_INPUT`**
- Severity: MEDIUM | Confidence: POSSIBLE | Temporality: HISTORICAL
- Adversary: passive_observer, kyc_exchange
- `⊂ CIOH`
- The transaction includes more inputs than the minimum subset needed to cover the payment plus estimated fee. Leaks coin selection algorithm behavior and the wallet's UTXO set composition.

**`DUST_SPENDING`**
- Severity: HIGH | Confidence: CERTAIN | Temporality: ACTIVE_RISK
- Adversary: state_adversary
- `→ CIOH`
- A transaction includes a dust input alongside one or more non-dust inputs. Actively links the dust probe to the wallet's other UTXOs.

**`TAINTED_UTXO_MERGE_L1`**
- Severity: HIGH | Confidence: CERTAIN (if source in database) | Temporality: HISTORICAL
- Adversary: kyc_exchange, state_adversary
- `→ CIOH`
- A wallet transaction included ≥ 1 input from a known tainted source AND ≥ 1 clean input. The wallet owner actively merged a flagged coin with clean funds.
- Escalation: `sanctions` or `darknet` source → CRITICAL.

**`SCRIPT_TYPE_MIXING`**
- Severity: MEDIUM | Confidence: CERTAIN | Temporality: HISTORICAL
- Adversary: passive_observer
- A transaction includes inputs from ≥ 2 distinct script families (P2PKH, P2SH, P2WPKH, P2WSH, P2TR, P2SH-P2WPKH). Mixed script inputs create a strong wallet fingerprint and shrink the anonymity set.

**`CHANGE_DETECTION`**
- Severity: MEDIUM | Confidence: POSSIBLE | Temporality: HISTORICAL
- Adversary: passive_observer, kyc_exchange
- The change output is identifiable through standard heuristics: round-number payment output, same script type as inputs, consistent fixed change output position across ≥ 3 transactions. Signals that the payment/change split is unambiguous to a passive observer.

**`OUTPUT_ORDERING_FINGERPRINT`**
- Severity: LOW | Confidence: PROBABLE | Temporality: ONGOING_PATTERN
- Adversary: passive_observer
- `→ BEHAVIORAL_FINGERPRINT`
- Outputs across wallet transactions are consistently ordered in the same deterministic pattern (lexicographically by scriptPubKey, by value ascending/descending, or change always at a fixed position). Consistent ordering identifies the wallet software. BIP69 specifies lexicographic ordering; detecting this pattern is the signal.

---

##### Tier 2 — Wallet-Local (multi-transaction)

**`BEHAVIORAL_FINGERPRINT`**
- Severity: HIGH | Confidence: PROBABLE | Temporality: ONGOING_PATTERN
- Adversary: passive_observer, kyc_exchange
- Rollup finding. Fires when ≥ 2 sub-signals (`FEE_RATE_FINGERPRINT`, `AMOUNT_PATTERN_FINGERPRINT`, `RBF_FINGERPRINT`, `OUTPUT_ORDERING_FINGERPRINT`) fire for the same wallet. Consistent behavioral patterns re-identify the wallet across chain scans even without address linkage.
- Escalation: ≥ 4 sub-signals → CRITICAL.

**`FEE_RATE_FINGERPRINT`**
- Severity: MEDIUM | Confidence: PROBABLE | Temporality: ONGOING_PATTERN
- Adversary: passive_observer
- `→ BEHAVIORAL_FINGERPRINT`
- Fee rates across wallet transactions have coefficient of variation < 0.15, or ≥ 80% of transactions (minimum 3 samples) use the same round sat/vbyte value.

**`AMOUNT_PATTERN_FINGERPRINT`**
- Severity: MEDIUM | Confidence: POSSIBLE | Temporality: ONGOING_PATTERN
- Adversary: passive_observer, kyc_exchange
- `→ BEHAVIORAL_FINGERPRINT`
- ≥ 60% of payment outputs (minimum 3 samples) are round BTC denominations (divisible by 10^k sat, k ≥ 3). Also flags "always no-change" spending (sends exact UTXO value).

**`RBF_FINGERPRINT`**
- Severity: LOW | Confidence: PROBABLE | Temporality: ONGOING_PATTERN
- Adversary: passive_observer
- `→ BEHAVIORAL_FINGERPRINT`
- Wallet consistently signals RBF (nSequence < 0xFFFFFFFE) or consistently does not (> 90% uniformity, minimum 3 transactions). Uniform RBF policy identifies the wallet software.

**`PEELING_CHAIN`**
- Severity: HIGH | Confidence: PROBABLE | Temporality: HISTORICAL
- Adversary: passive_observer, kyc_exchange, state_adversary
- Escalation: chain length ≥ 5 hops → CRITICAL
- The wallet's own spending forms a peeling chain: a sequence of transactions each with exactly 1 input and exactly 2 outputs where the wallet is the spender in each hop, forming a chain ≥ 3 hops. The wallet progressively peels off payments from a single coin, leaving a trivially traceable spending trail.
- Requires `txindex=1`.

**`UTXO_AGE_SPREAD`**
- Severity: LOW | Confidence: CERTAIN | Temporality: HISTORICAL
- Adversary: kyc_exchange, state_adversary
- The block height spread across co-spent UTXOs exceeds a configurable threshold (default: 52,560 blocks ≈ 1 year). Reveals dormancy and lookback patterns.

**`EXCHANGE_ORIGIN`**
- Severity: MEDIUM | Confidence: POSSIBLE (structural) / CERTAIN (database) | Temporality: HISTORICAL
- Adversary: kyc_exchange, state_adversary
- The funding transaction matches exchange batch-withdrawal patterns (≥ 5 outputs to distinct addresses, output amounts in exchange-typical ranges), or a known exchange address appears in the funding transaction per the labeled-address database. Exchanges perform KYC; this UTXO may be linked to a real identity at the source.

---

##### Tier 2 — Taint (multi-level propagation)

**`DIRECT_TAINT`** *(warning)*
- Severity: varies | Confidence: CERTAIN | Temporality: ACTIVE_RISK
- Adversary: kyc_exchange, state_adversary
- UTXO was received directly from a known-tainted source. The UTXO is unspent; risk is prospective. Severity by source type: `exchange` → MEDIUM, `mixer` → HIGH, `darknet`/`sanctions` → CRITICAL.

**`TAINTED_UTXO_MERGE_L1`**
- Severity: HIGH | Confidence: CERTAIN | Temporality: HISTORICAL
- Adversary: kyc_exchange, state_adversary
- `→ CIOH`
- *(see Tier 1 above)*

**`TAINTED_UTXO_MERGE_L2`**
- Severity: MEDIUM | Confidence: PROBABLE | Temporality: HISTORICAL
- Adversary: kyc_exchange, state_adversary
- A received UTXO came from a transaction that itself included an L1-tainted input. Indirect exposure — someone else merged tainted coins before sending to the wallet.

**`TAINTED_UTXO_MERGE_LN`** (N = 3–5)
- Severity: LOW | Confidence: POSSIBLE | Temporality: HISTORICAL
- Adversary: state_adversary
- Taint propagated N hops back. Proportional (haircut) accounting: at each merge hop, `taint_fraction = tainted_input_value / total_input_value`; fractions multiply across hops. At fan-out, each output inherits the same taint fraction as the transaction. Finding fires if accumulated taint fraction > configurable threshold (default: 10%). Beyond 5 hops: annotate rather than flag.

**Taint accounting model**: proportional (haircut) by default. Taint dilutes multiplicatively at each hop in proportion to the contaminated fraction of inputs. This differs from the poison model used by Chainalysis. The model is documented and configurable.

---

##### Tier 3 — Graph Traversal

**`CLUSTER_MERGE`**
- Severity: CRITICAL | Confidence: PROBABLE | Temporality: HISTORICAL
- Adversary: passive_observer, kyc_exchange, state_adversary
- `⊃ CIOH`
- Inputs in a transaction come from ≥ 2 previously disconnected address clusters, where cluster membership is inferred by multi-hop CIOH analysis across the full chain. This merges previously separate identity clusters into one traceable entity.
- Requires `txindex=1`. Suppresses `CIOH` score contribution when it fires.
- Cluster graph is keyed by **address** (not UTXO outpoint). Multiple UTXOs at the same address are the same cluster node by definition.

---

##### Warnings

**`DORMANT_UTXOS`** *(warning)*
- Temporality: ACTIVE_RISK
- One or more wallet UTXOs have not moved for an extended period (default threshold: 1 year). Not scored. Contextual signal flagged for user awareness.

### Detection Algorithms

#### External Data: Labeled-Address Database

Taint and exchange-origin detectors depend on a labeled-address database:

- Shipped as a versioned file (`data/labels.json`) bundled in the binary release
- User can augment by placing additional files in a configured directory; all files merged at startup
- **No network calls at any time** — freshness is a documentation concern, not a runtime concern

Schema per address entry:

```json
{
  "address": "bc1q...",
  "label": "Binance",
  "type": "exchange",
  "confidence": "high",
  "source": "community"
}
```

Schema for cluster-level entries (pattern-based, no specific address):

```json
{
  "label": "Binance Withdrawal Cluster",
  "type": "exchange",
  "cluster_fingerprint": {
    "min_outputs_per_tx": 5,
    "dominant_script_type": "p2wpkh",
    "fee_rate_range_sat_vb": [1, 8]
  }
}
```

Types: `exchange | sanctions | darknet | mixer | custom`

Known limitation: address-level tagging may over-attribute taint when a single address has mixed-provenance UTXOs (e.g., a specific withdrawal from a sanctioned exchange, not all outputs from that address).

#### Per-Finding Algorithm Sketches

**`ADDRESS_REUSE`**: Count distinct txids funding each descriptor-derived address. If > 1, flag. Deterministic. Exception: BIP352 (Silent Payments) outputs use a unique address per sender by protocol design — suppress this finding for valid Silent Payment addresses.

**`CHANGE_REUSE`**: For each outgoing transaction, identify the change output by priority: (1) BIP32 internal derivation path if descriptor-derived, (2) same script type as inputs with non-round amount, (3) inconclusive. If the change address was previously funded externally, flag. Confidence degrades to POSSIBLE when change identification is inconclusive.

**`DUST`**: For each UTXO (current and historical), check `value < dust_threshold[script_type]`. Flag even if already spent (temporal tag: HISTORICAL).

**`CIOH`**: For each wallet transaction with ≥ 2 wallet-owned inputs, flag. CoinJoin exception: suppress if ≥ 3 equal-value outputs AND ≥ 3 distinct non-wallet inputs.

**`CONSOLIDATION`**: CIOH fires AND input count ≥ 3 AND output count ≤ 2.

**`UNNECESSARY_INPUT`**: (1) Estimate payment amount as largest non-change output. (2) Compute `needed = payment_amount + estimated_fee`. (3) Find minimum input subset covering `needed`. (4) If actual input count > minimum + 1, flag. Confidence: POSSIBLE. If change identification is inconclusive (no BIP32 path, both outputs similar value), suppress entirely — false positive risk outweighs signal.

**`DUST_SPENDING`**: For each wallet transaction: check if any input's value at UTXO creation time was below dust threshold AND at least one other input is above threshold.

**`TAINTED_UTXO_MERGE_L1`**: For each wallet transaction: check if any input address appears in the labeled-address database. If yes and at least one clean input was co-spent, flag.

**`TAINTED_UTXO_MERGE_L2/LN`**: Traverse backwards from each wallet UTXO through its funding transaction's inputs. At each hop, compute `taint_fraction = tainted_input_value / total_input_value`. Propagate multiplicatively. Flag if taint_fraction > threshold at hop N. Maximum depth: 5.

**`DIRECT_TAINT`**: For each unspent UTXO, identify the sender: the address(es) that provided inputs to the transaction that created this UTXO. Check if any sender address appears in the labeled-address database. Distinct from `TAINTED_UTXO_MERGE_L1`: here the wallet received funds directly from a tainted entity (no co-spend of clean inputs).

**`SCRIPT_TYPE_MIXING`**: For each wallet transaction, collect script families across all inputs. Flag if distinct families ≥ 2.

**`CHANGE_DETECTION`**: For each wallet transaction: check (a) payment output is a round denomination, (b) change output shares script type with inputs, (c) wallet consistently places change at the same output position (always index 0 or always last) across ≥ 3 transactions. Flag if ≥ 2 of 3 heuristics agree.

**`OUTPUT_ORDERING_FINGERPRINT`**: For each wallet transaction, test if outputs are sorted lexicographically by scriptPubKey or by value. Accumulate pattern counts. Flag if ≥ 80% of outgoing transactions follow the same deterministic ordering, with ≥ 3 sample transactions.

**`FEE_RATE_FINGERPRINT`**: Compute fee rate (sat/vbyte) for all outgoing wallet transactions. Flag if coefficient of variation < 0.15 OR ≥ 80% use the same round sat/vbyte value (minimum 3 samples).

**`AMOUNT_PATTERN_FINGERPRINT`**: For each spend, check if payment output is a round BTC denomination (value % 10^k = 0, k ≥ 3). Flag if ≥ 60% of spends match (minimum 3 samples).

**`RBF_FINGERPRINT`**: Check nSequence across all wallet transaction inputs. Flag if consistently signals RBF or consistently does not (> 90% uniformity, ≥ 3 transactions).

**`BEHAVIORAL_FINGERPRINT`** (rollup): Fires when ≥ 2 sub-signals fire for the same wallet. Severity: HIGH base, CRITICAL at ≥ 4 sub-signals.

**`PEELING_CHAIN`**: For each wallet-initiated outgoing transaction with exactly 1 input and exactly 2 outputs: treat as hop 1. Follow the subsequent wallet-initiated spends of the same coin — each hop must also be a wallet-controlled transaction with exactly 1 input and exactly 2 outputs. Flag if chain reaches ≥ 3 consecutive wallet-initiated hops. Requires `txindex=1`.

**`UTXO_AGE_SPREAD`**: For UTXOs co-spent in a transaction, compute `max(creation_height) - min(creation_height)`. Flag if > threshold (default: 52,560 blocks).

**`EXCHANGE_ORIGIN`**: Tier 1 (structural): funding tx has ≥ 5 outputs to distinct addresses with outputs in exchange-typical ranges → POSSIBLE. Tier 2 (database): any input address in funding tx matches labeled-address database with `type: exchange` → CERTAIN.

**`CLUSTER_MERGE`**: Requires full-chain Union-Find cluster graph. For each wallet transaction with ≥ 2 inputs: check if any two inputs belong to different cluster IDs before this transaction. If yes, flag.

### Privacy Scoring Model

#### Per-Finding Contribution

```
contribution(finding) = severity_weight × confidence_multiplier
```

| Severity | Weight | Confidence | Multiplier |
|----------|--------|------------|------------|
| CRITICAL | 25 | CERTAIN | 1.0 |
| HIGH | 15 | PROBABLE | 0.7 |
| MEDIUM | 8 | POSSIBLE | 0.4 |
| LOW | 3 | | |

#### Deduplication

For a UTXO's fired finding set F:

1. For each `(A ⊂ B)` pair where both A, B ∈ F: remove B (suppress general; keep specific)
2. For each `(A ⊃ B)` pair where both A, B ∈ F: remove B (suppress subsumed; keep richer)
3. Score the remaining set

In both rules, A is always kept and B is always removed. `→` (causal) edges are never deduplicated.

#### Per-UTXO Score

```
utxo_score = Σ contribution(f) for f in deduplicated_findings(utxo)
```

#### Wallet-Level Privacy Exposure Score

```
wallet_score = min(100, Σ utxo_scores + wallet_level_bonuses)
```

Wallet-level findings (`BEHAVIORAL_FINGERPRINT`, `PEELING_CHAIN`, `UTXO_AGE_SPREAD`) are scored exclusively in `wallet_level_bonuses` and are never included in per-UTXO scores. Each finding is counted in exactly one of the two terms.

The score is a benchmark of current UTXO privacy state. Decision-making support (which UTXOs to spend together, remediation prioritization) is deferred to a future SIP; the score is intentionally not decomposed into actionability tiers in this version.

#### Severity Escalation

| Finding | Condition | Escalated Severity |
|---------|-----------|-------------------|
| `PEELING_CHAIN` | chain length ≥ 5 hops | CRITICAL |
| `TAINTED_UTXO_MERGE_L1` | source type = sanctions / darknet | CRITICAL |
| `DIRECT_TAINT` | source type = sanctions / darknet | CRITICAL |
| `BEHAVIORAL_FINGERPRINT` | ≥ 4 sub-signals fire | CRITICAL |

### Cluster Graph Architecture

Full-chain `CLUSTER_MERGE` requires a persistent Union-Find data structure over all CIOH-linkable transactions on chain:

- **Storage**: SQLite (default) or RocksDB (configurable)
- **Initial sync**: full chain scan on first run — documented as potentially hours on mainnet; progress reported to user
- **Incremental updates**: subsequent runs process only blocks since last sync height
- **Interface**: `cluster_id(address) → ClusterId`, `same_cluster(addr_a, addr_b) → bool` — implementation is swappable
- **Index key**: addresses (not UTXO outpoints). CIOH operates at the address level; multiple UTXOs at the same address are the same cluster node.
- **Degraded mode**: when `txindex=1` is unavailable, a partial cluster graph is built from the wallet's own transaction history only. Results are incomplete but correct for visible transactions.

Bitcoin Core requirements:

| Requirement | Purpose |

### PSBT Pre-Flight Analysis *(Planned: Phase 2)*

A key feature for wallet integrations: before a user broadcasts a transaction, Stealth can analyze the unsigned PSBT and report privacy implications:

- Would this transaction create a CIOH link between previously unlinked UTXOs?
- Is the change output trivially detectable?
- Does the transaction mix script types?
- Would spending this set of inputs merge clusters?
- Is the fee rate an outlier that fingerprints the wallet?

The pre-flight API accepts a PSBT (base64-encoded) and returns findings with severity, allowing the wallet to display warnings before confirmation. This feature is deferred to Phase 2 (REST API) and is out of scope for the initial CLI release.

### Remediation Engine

Stealth diagnoses and prescribes. Each finding type maps to one or more remediation strategies:

- **`ADDRESS_REUSE` / `CHANGE_REUSE`**: Generate and use fresh addresses. For wallets supporting BIP352 (Silent Payments), recommend migration to eliminate reuse permanently.
- **`CIOH` / `CLUSTER_MERGE`**: Recommend spending UTXOs from different clusters in separate transactions. Suggest CoinJoin or payjoin if supported.
- **`DUST` / `DUST_SPENDING`**: Flag dust UTXOs for "do not spend" in coin control. If already co-spent, note the damage and suggest mixing the tainted output.
- **`CHANGE_DETECTION`**: Recommend wallets that randomize change position and use same-type change outputs. Suggest spending exact amounts when possible.
- **`SCRIPT_TYPE_MIXING`**: Migrate all funds to a single script type (P2TR preferred for uniformity).
- **`EXCHANGE_ORIGIN`**: Recommend a CoinJoin or time-delayed self-transfer to break the exchange linkage.
- **`PEELING_CHAIN`**: Consolidate the peel chain output via CoinJoin before reuse; avoid single-input-two-output spend patterns.
- **`BEHAVIORAL_FINGERPRINT` sub-signals**: Addressable individually — change fee estimation method, randomize change position, adjust RBF policy.

Remediations are suggestions, not actions. Stealth never constructs, signs, or broadcasts transactions.

### Silent Payments Awareness (BIP352)

BIP352 (Silent Payments) was updated to v1.1.0 (March 2, 2026) and is being adopted by Bitcoin Core 28.0+, Cake Wallet, Silentium, and BlueWallet. Silent Payments fundamentally address `ADDRESS_REUSE` by deriving unique on-chain addresses from a single static payment code.

Stealth handles BIP352 in two ways:

1. **False-positive suppression**: Recognize P2TR outputs generated by the BIP352 protocol and exclude them from `ADDRESS_REUSE` findings — repeated funding of the same Silent Payment address is expected and non-leaking.
2. **Remediation recommendation**: Recommend Silent Payments as the remediation for `ADDRESS_REUSE` and `CHANGE_REUSE` findings when the user's wallet supports BIP352.

### Explicit Out of Scope

| Excluded | Reason |
|----------|--------|
| Lightning Network / off-chain | Different privacy model; no on-chain UTXO state |
| Key management / signing | Stealth is strictly read-only |
| Real-time mempool surveillance | Analyzes settled state only |
| Cross-chain tracking | No UTXO model parity |
| Side-channel attacks | Different tool class |
| CoinJoin coordination internals | Detects absence of mixing, not mixing quality |
| Protocol-level privacy improvements | Reports findings; does not prescribe wallet or protocol solutions |
| External API calls of any kind | Core security model: no network surface beyond user's node |
| Automated remediation / transaction construction | Report only; never build or sign transactions |
| PSBT pre-flight analysis | Deferred to Phase 2 |
| Alternative node backends (Electrum, Esplora) | Deferred — Bitcoin Core JSON-RPC is the baseline |

## Deployment

### Core Library and CLI

- Rust rewrite of all detectors from `detect.py` as `stealth-core` crate.
- CLI binary (`stealth-cli`): accepts a descriptor, connects to Bitcoin Core RPC, outputs JSON or human-readable report.
- Test vectors generated from the existing Python `reproduce.py` regtest scenarios.
- CI/CD with cross-compilation (linux-amd64, linux-arm64, macos-amd64, macos-arm64).
- Published to crates.io under BSD-3-Clause.

### BDK Integration

- `stealth-bdk` crate with a `PrivacyAnalyzer` trait compatible with `bdk_wallet::Wallet`.
- PSBT pre-flight as a `TxBuilder` extension: `tx_builder.privacy_check()` returns warnings before `finish()`.
- Privacy-aware coin selection: a custom `CoinSelectionAlgorithm` that penalizes selections creating CIOH links, cluster merges, or script-type mixing.

BDK powers: Bitkey, Peach Bitcoin, Envoy (Foundation), Bull Bitcoin, Padawan Wallet, Lava. This phase has the highest integration leverage.

### REST API and Web UI

- Lightweight HTTP API server (Axum) wrapping `stealth-core`.
- Endpoints: `/api/scan`, `/api/score`, `/api/preflight`, `/api/export`.
- PSBT pre-flight analysis endpoint (pre-broadcast privacy warning).
- React-based web UI connecting to the local API.
- Docker image and Docker Compose template for standalone deployment.

### Umbrel and Start9 App Store

- Umbrel app package (`umbrel-app.yml`, `docker-compose.yml`) following the Umbrel App Framework.
- Auto-configuration via Umbrel's `$APP_BITCOIN_*` environment variables.
- Start9 service wrapper with StartOS manifest.
- Submission to both app stores.

This addresses [Issue #4](https://github.com/LORDBABUINO/stealth/issues/4).

### Sparrow Wallet Integration

Sparrow is Java-based and does not expose a plugin API. Three integration paths, in order of preference:

1. **Upstream contribution**: "Privacy Audit" tab contributed to Sparrow's codebase, calling Stealth's REST API or a JNI-wrapped `stealth-core`. Requires coordination with craigraw.
2. **Companion mode**: Stealth runs alongside Sparrow, connecting to the same node. User exports a descriptor from Sparrow and imports into Stealth.
3. **PSBT interop**: Sparrow exports a PSBT; Stealth's pre-flight API analyzes and returns warnings before signing.

Approach (1) is the ideal end state. Approach (2) is the pragmatic starting point. Approach (3) is a quick win with minimal integration effort.

### Language Bindings and Mobile

- Python bindings via PyO3 (`stealth-py` on PyPI).
- Kotlin/Java bindings via JNI (`stealth-jvm` on Maven Central).
- Swift bindings via UniFFI for iOS wallet integration.
- WASM build for browser-based analysis (descriptor + UTXO set mode, no node connection required).

### Future Research

- **Floresta backend**: Investigate Floresta's Electrum server as a lightweight alternative for resource-constrained devices.
- **Anon-set estimation**: Estimate actual anonymity set size per UTXO rather than binary found/not-found per heuristic.
- **Temporal analysis**: Detect timing-based fingerprints (consistent broadcast time suggesting timezone).
- **CoinJoin quality scoring**: Assess mix quality for UTXOs that went through a CoinJoin (equal outputs, participant count, post-mix spending behavior).
- **Taproot script-path privacy**: Analyze how key-path vs. script-path spending reveals multisig or timelock conditions.
- **Compact block filter scanning (BIP158)**: Enable descriptor scanning without `txindex` using BIP158 filters to identify relevant blocks.

## Technical Decisions and Tradeoffs

### Why not use external APIs?

External APIs violate Stealth's core privacy guarantee. Querying mempool.space or blockchain.info with your addresses reveals your wallet to that service. Every chain query goes exclusively to the user's own node.

### Why descriptor-native?

Descriptors are the modern standard for wallet spending conditions. BDK is descriptor-native. Bitcoin Core's wallet has moved to descriptors. Sparrow uses them internally. Descriptor input supports arbitrary policies (single-sig, multisig, Taproot, timelocked) without special-casing each script type.

### Why full-chain CLUSTER_MERGE over wallet-local?

Wallet-local cluster analysis only finds links visible within the wallet's own transaction history. Full-chain CLUSTER_MERGE finds links created by any party on chain — including coinjoins by third parties that happen to reuse addresses, or exchange hot wallets that merged with the user's UTXO history. The full-chain version requires `txindex=1` and a significant initial sync, but produces results qualitatively closer to what a chain analyst actually sees. The degraded mode (wallet-local only) is always available as a fallback.
